# Git 分支管理与 GitHub Pages 部署实战
#### March 11, 2026

> 在现代 Web 开发中，Git 分支管理和自动化部署是每个开发者必备的技能。本文将分享我在实际项目中使用 Git 分支策略和 GitHub Pages 部署静态网站的经验。

## 为什么需要分支管理

在开发过程中，合理的分支策略可以帮助我们：

- **隔离开发环境** - 新功能开发不影响主分支稳定性
- **并行开发** - 多个功能可以同时开发
- **版本控制** - 轻松回滚到历史版本
- **代码审查** - 通过 Pull Request 进行代码审核

## 我的分支策略

在我的博客项目中，我使用了以下分支结构：

### master 分支
主开发分支，包含完整的源代码和构建配置。

### gh-pages 分支
专门用于 GitHub Pages 部署，只包含构建后的静态文件。

### develop 分支
日常开发分支，新功能在这里开发和测试。

### 功能分支
临时分支，命名格式如 `feature/xxx` 或 `fix/xxx`。

## GitHub Pages 部署流程

```bash
# 1. 在 master 分支完成开发
git checkout master

# 2. 构建静态文件
npm run build

# 3. 切换到 gh-pages 分支
git checkout gh-pages

# 4. 更新部署文件
# 将构建产物复制到对应目录

# 5. 提交并推送
git add .
git commit -m "deploy: 更新博客内容"
git push origin gh-pages
```

## 自动化部署的优势

使用 GitHub Pages 部署静态网站有以下好处：

1. **免费** - GitHub 提供免费托管服务
2. **简单** - 推送到 gh-pages 分支即可自动部署
3. **HTTPS** - 自动启用 HTTPS 加密
4. **自定义域名** - 支持绑定自己的域名

## 常见问题与解决方案

### 问题 1：部署后页面没有更新

**原因**：浏览器缓存

**解决**：
- 强制刷新页面 (Ctrl+F5)
- 在文件名中添加版本号或时间戳

### 问颲 2：分支合并冲突

**原因**：多个分支修改了同一文件

**解决**：
- 定期同步主分支到功能分支
- 使用 `git rebase` 保持提交历史整洁

## 最佳实践

1. **分支命名规范** - 使用清晰的命名，如 `feature/login-page`
2. **小步提交** - 每次提交只做一件事
3. **提交信息清晰** - 使用约定式提交规范
4. **及时合并** - 功能完成后尽快合并到主分支
5. **保护主分支** - 设置分支保护规则，禁止直接 push

## 总结

合理的 Git 分支管理和自动化部署流程可以大大提高开发效率。对于静态网站项目，GitHub Pages 是一个简单而强大的选择。
